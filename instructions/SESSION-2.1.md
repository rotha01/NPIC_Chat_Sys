### `laravel-app/app/Models/User.php`

> **Edited manually** — add the `HasApiTokens` trait.

```php
<?php

namespace App\Models;

// use Illuminate\Contracts\Auth\MustVerifyEmail;
use Database\Factories\UserFactory;
use Illuminate\Database\Eloquent\Attributes\Fillable;
use Illuminate\Database\Eloquent\Attributes\Hidden;
use Illuminate\Database\Eloquent\Factories\HasFactory;
use Illuminate\Foundation\Auth\User as Authenticatable;
use Illuminate\Notifications\Notifiable;
use Laravel\Sanctum\HasApiTokens;

#[Fillable(['name', 'email', 'password'])]
#[Hidden(['password', 'remember_token'])]
class User extends Authenticatable
{
    /** @use HasFactory<UserFactory> */
    use HasFactory, Notifiable, HasApiTokens;

    /**
     * Get the attributes that should be cast.
     *
     * @return array<string, string>
     */
    protected function casts(): array
    {
        return [
            'email_verified_at' => 'datetime',
            'password' => 'hashed',
        ];
    }
}
```

### `laravel-app/app/Http/Controllers/API/AuthController.php`

> **Generated by command, then manually edited** — the command creates the file and the `API/` subdirectory; paste the block below to replace its contents with the final version.

```bash
# Inside the laravel container
php artisan make:controller API/AuthController
```

```php
<?php

namespace App\Http\Controllers\API;

use App\Http\Controllers\Controller;
use App\Models\User;
use Illuminate\Http\Request;
use Illuminate\Support\Facades\Hash;
use Illuminate\Validation\ValidationException;

class AuthController extends Controller
{
    function signup(Request $request)
    {
        $request->validate([
            'name' => 'required|string|max:100',
            'email' => 'required|email|unique:users,email',
            'password' => 'required|string|min:6|max:10|confirmed'
        ]);

        $user = User::create([
            'name' => $request->name,
            'email' => $request->email,
            'password' => $request->password,
        ]);

        return response([
            'message' => 'User signed up.',
            'user' => $user
        ], 201);
    }

    function signin(Request $request)
    {
        $request->validate([
            'email' => 'required|email|exists:users,email',
            'password' => 'required|string|min:6|max:10'
        ]);

        $user = User::where('email', $request->email)->first();

        if (!Hash::check($request->password, $user->password)) {
            throw ValidationException::withMessages([
                'password' => 'Password does not match.',
            ]);
        }

        $token = $user->createToken('auth_token')->plainTextToken;

        return response([
            'message' => 'User signed in.',
            'user' => $user,
            'token' => $token
        ], 200);
    }

    function signout(Request $request)
    {
        $user = $request->user();

        // option 1
        $user->currentAccessToken()->delete();

        // option 2
        $currentToken = $user->currentAccessToken();
        $user->tokens()->where('id', $currentToken->id)->delete();

        return response([
            'message' => 'User signed out.'
        ], 200);
    }

    function verify(Request $request)
    {
        return response([
            'message' => 'Token is valid.',
            'user' => $request->user()
        ], 200);
    }
}
```

## How Each File Works

### `php artisan install:api` — what it generates

`php artisan install:api` is a Laravel 11+ convenience command that wires up API authentication in one step instead of requiring four manual operations. It:

- Adds Sanctum to `composer.json`/`composer.lock` via `composer require laravel/sanctum`.
- Publishes `config/sanctum.php` and the PAT migration (same result as `php artisan vendor:publish --provider="Laravel\Sanctum\SanctumServiceProvider"`).
- Edits `bootstrap/app.php` to register `routes/api.php`.
- Creates a minimal `routes/api.php` stub.

The entrypoint script runs `composer install` on every container start, so Sanctum is available automatically on the next `docker compose up`.

---

### `laravel-app/config/sanctum.php` + PAT migration

Both files are published by `php artisan install:api`. Key settings in `config/sanctum.php` for this project:

- **`stateful`** — lists domains that can use cookie-based session authentication (for SPAs). Since this project uses stateless Bearer token auth from the Vue.js frontend, this list is not directly exercised by the current routes, but it is required by the config.
- **`guard`** — falls back to `['web']` when no Bearer token is present. Irrelevant for pure API token flows.
- **`expiration`** — set to `null`, meaning tokens never expire automatically. Set to a number of minutes to enable time-based expiry.
- **`token_prefix`** — empty by default. Set `SANCTUM_TOKEN_PREFIX` in `.env` to prefix all issued tokens (useful for GitHub-style secret scanning detection).

---

### `laravel-app/bootstrap/app.php`

The `api:` line was added automatically by `php artisan install:api`. It registers `routes/api.php` with the Laravel router. Without this, every route defined in `api.php` would be silently ignored — the file would exist but never be loaded.

Laravel's `withRouting()` helper automatically applies the `api` middleware group (which includes `throttle:api` rate limiting and `SubstituteBindings`) to all routes in `api.php`. It also prefixes every route URL with `/api/`, so `Route::post('/signup', ...)` becomes reachable at `http://localhost:8000/api/signup`.

---

### `laravel-app/app/Models/User.php`

Adding `HasApiTokens` to the `use` statement mixes in three key methods from Sanctum:

- `createToken(string $name)` — inserts a row into `personal_access_tokens` and returns a `NewAccessToken` object. Calling `.plainTextToken` on it gives the raw token string (shown to the user once; only a SHA-256 hash is stored in the database).
- `currentAccessToken()` — returns the `PersonalAccessToken` model instance that authenticated the current request.
- `tokens()` — a polymorphic `HasMany` relation to all tokens belonging to the user.

The `#[Fillable]` and `#[Hidden]` PHP 8 attributes on the class are equivalent to setting the `$fillable` and `$hidden` array properties — they declare which columns may be mass-assigned and which are stripped from JSON responses (`password` and `remember_token` are never sent to the client).

The `casts()` method tells Laravel to automatically hash any value assigned to `password` before saving it to the database (via the `hashed` cast). This means `User::create(['password' => 'plaintext'])` stores the bcrypt hash, not the plain string.

---

### `laravel-app/app/Http/Controllers/API/AuthController.php`

The controller lives in the `App\Http\Controllers\API` namespace, corresponding to the `app/Http/Controllers/API/` directory. Laravel does not generate this subdirectory automatically — it was created manually.

**`signup()`**

1. Validates the incoming JSON body. `confirmed` on `password` means the request must also include a `password_confirmation` field with the same value.
2. Calls `User::create()`. Because `password` has the `hashed` cast, the plain-text value is bcrypt-hashed before the INSERT.
3. Returns a `201 Created` response with the new user record. The `password` and `remember_token` fields are stripped by the `#[Hidden]` attribute before serialisation.

**`signin()`**

1. Validates the request (same password length constraints as signup for consistency).
2. Looks up the user by email. If not found, or if `Hash::check()` fails (plain password vs. stored hash), returns `401 Unauthorized`. Using the same generic error message for both cases prevents user enumeration.
3. Calls `createToken('auth_token')` — the string `'auth_token'` is a human-readable label stored in the `name` column of `personal_access_tokens`. It does not affect behaviour.
4. Returns the plain-text token in the response body. **This is the only time the raw token is visible** — Sanctum stores only the hash from this point on.

**`signout()`**

The method shows two equivalent approaches to deleting the current token. In practice, you only need one:

- **Option 1** (`$user->currentAccessToken()->delete()`) — retrieves the token model from the current request context and deletes it immediately. Simplest form.
- **Option 2** — retrieves the token model, then issues a query-builder `DELETE` scoped by the token's ID. Functionally identical. Useful if you need the token object for other logic before deleting it.

Because both options run sequentially in the current code, option 2 attempts to delete a row that option 1 has already removed. This is harmless (the query just matches zero rows), but in production you would keep only one approach.

---

### `laravel-app/database/migrations/2026_05_09_063841_create_personal_access_tokens_table.php`

Scaffolded by `php artisan vendor:publish` (see above). Sanctum requires this table to store issued tokens. Key columns:

| Column | Type | Purpose |
|---|---|---|
| `tokenable_type` + `tokenable_id` | polymorphic morph | Links the token to any model (here: `User`). Allows tokens for multiple model types in one table. |
| `name` | text | Human-readable label set at token creation time (e.g., `'auth_token'`). |
| `token` | varchar(64) | SHA-256 hash of the raw token. Never the plain value. |
| `abilities` | text (JSON) | Scopes for the token (e.g., `["read", "write"]`). Unused in this session. |
| `last_used_at` | timestamp | Updated by Sanctum on each authenticated request. |
| `expires_at` | timestamp (indexed) | Optional expiry; `null` means the token never expires. |

The timestamp in the filename (`2026_05_09_063841`) is generated at the moment `vendor:publish` runs and determines migration order — this migration runs after the users table.

---

### `laravel-app/routes/api.php`

All routes in this file are automatically prefixed with `/api/` and wrapped in the `api` middleware group by the registration in `bootstrap/app.php`.

- `/api/signup` and `/api/signin` are open routes — no authentication middleware, so anyone can call them.
- `/api/signout` has `->middleware('auth:sanctum')`, meaning Laravel will:
  1. Read the `Authorization: Bearer <token>` header.
  2. Hash the token and look it up in `personal_access_tokens`.
  3. Resolve the owning `User` model and attach it to the request as `$request->user()`.
  4. Return `401 Unauthenticated` automatically if the token is missing, invalid, or expired — before the controller method is ever called.

## How Sanctum Token Authentication Works

```
Client                          Laravel
  │                               │
  │  POST /api/signin             │
  │  { email, password }  ───────►│  Hash::check(password, hash)
  │                               │  $user->createToken('auth_token')
  │                               │    → INSERT INTO personal_access_tokens
  │                               │      (stores SHA-256 hash only)
  │◄─── { token: "abc|xyz..." } ──│
  │                               │
  │  POST /api/signout            │
  │  Authorization: Bearer abc|xyz│
  │  ──────────────────────────► │  Sanctum hashes "abc|xyz"
  │                               │  → SELECT from personal_access_tokens
  │                               │  → resolves $request->user()
  │                               │  → controller runs signout()
  │                               │  → token row deleted
  │◄── { message: "signed out" } ─│
```

The token returned by `createToken()->plainTextToken` has the format `{id}|{randomBytes}`. The `id` is the primary key of the row in `personal_access_tokens`. Sanctum splits on `|`, looks up the row by `id`, and then verifies the rest against the stored SHA-256 hash.

## API Endpoints

### `POST /api/signup`

**Request body (JSON):**
```json
{
  "name": "Alice",
  "email": "alice@example.com",
  "password": "secret1",
  "password_confirmation": "secret1"
}
```

**Success response — `201 Created`:**
```json
{
  "message": "User signed up.",
  "user": {
    "id": 1,
    "name": "Alice",
    "email": "alice@example.com",
    "email_verified_at": null,
    "created_at": "2026-05-09T06:38:41.000000Z",
    "updated_at": "2026-05-09T06:38:41.000000Z"
  }
}
```

**Validation failure — `422 Unprocessable Entity`** (example: email already taken):
```json
{
  "message": "The email has already been taken.",
  "errors": {
    "email": ["The email has already been taken."]
  }
}
```

---

### `POST /api/signin`

**Request body (JSON):**
```json
{
  "email": "alice@example.com",
  "password": "secret1"
}
```

**Success response — `200 OK`:**
```json
{
  "message": "User signed in.",
  "user": { ... },
  "token": "1|abc123randomtokenstring..."
}
```

**Invalid credentials — `401 Unauthorized`:**
```json
{
  "message": "Invalid credentials."
}
```

---

### `POST /api/signout`

**Headers:**
```
Authorization: Bearer 1|abc123randomtokenstring...
```

**Success response — `200 OK`:**
```json
{
  "message": "User signed out."
}
```

**Missing or invalid token — `401 Unauthorized`** (returned automatically by Sanctum, before the controller runs):
```json
{
  "message": "Unauthenticated."
}
```

## Troubleshooting

### `POST /api/signup` returns `404 Not Found`

The API routes file is not registered. Verify `bootstrap/app.php` contains `api: __DIR__.'/../routes/api.php'` inside `->withRouting(...)`. Without this line, `routes/api.php` is never loaded regardless of its contents.

### `POST /api/signin` returns `500` with "Class not found: HasApiTokens"

`composer install` has not been run since `laravel/sanctum` was added to `composer.json`. Shell into the container and run `composer install`, or restart the stack so the entrypoint script re-runs it automatically.

### `POST /api/signout` returns `401 Unauthenticated`

The `Authorization` header is missing or malformed. The correct format is:
```
Authorization: Bearer <token>
```
Also ensure the token has not already been revoked (a sign-out deletes the token row — the same token cannot be reused).

### `php artisan migrate` fails with "Table 'personal_access_tokens' already exists"

The migration was run before on the same database. Sanctum's migration is idempotent only when the table does not exist. Either run `php artisan migrate:fresh` (drops all data) or drop just that table:
```bash
docker exec -it mysql-container mysql -u chat_user -pchat_password chat_system \
  -e "DROP TABLE IF EXISTS personal_access_tokens;"
php artisan migrate
```

### `signup` succeeds but `password` appears in the JSON response

The `#[Hidden]` attribute on the `User` model must list `'password'`. If you are serialising the user manually (e.g., `$user->toArray()`), call `$user->makeHidden('password')` explicitly. Using `response(['user' => $user])` respects `#[Hidden]` automatically.

### Tokens are not expiring

`expiration` is set to `null` in `config/sanctum.php`. To expire tokens after N minutes, set `'expiration' => N` and run the Sanctum token pruning command on a schedule:
```bash
php artisan sanctum:prune-expired --hours=24
```
