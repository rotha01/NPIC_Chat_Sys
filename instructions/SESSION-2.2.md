## What Changed in Session 2.2

Session 2.1 added Sanctum token authentication with inline validation and raw model responses. Session 2.2 refactors the controller by extracting validation into dedicated Form Request classes and wrapping user output through an API Resource.

| Area | Session 2.1 | Session 2.2 |
|---|---|---|
| `signup()` validation | `$request->validate([...])` inside controller | Extracted to `SignupRequest` Form Request |
| `signin()` validation | `$request->validate([...])` inside controller | Extracted to `SigninRequest` Form Request |
| User response shape | Raw `$user` Eloquent model | `new UserResource($user)` — explicit field selection |
| `AuthController` dependencies | `Illuminate\Http\Request` | `SignupRequest`, `SigninRequest`, `UserResource` |
| New files | — | `SignupRequest.php`, `SigninRequest.php`, `UserResource.php` |
| API routes | Unchanged | Unchanged |
| Database schema | Unchanged | Unchanged |

The three new classes are scaffolded with Artisan commands (`make:request`, `make:resource`), then edited manually to add the actual validation rules and field definitions. `AuthController.php` already exists from Session 2.1 and is edited manually to use those new classes.

### `laravel-app/app/Http/Controllers/API/AuthController.php`

> **Edited manually** — the file already exists from Session 2.1; paste the block below to replace its contents with the Session 2.2 version.

```php
<?php

namespace App\Http\Controllers\API;

use App\Http\Controllers\Controller;
use App\Http\Requests\User\SigninRequest;
use App\Http\Requests\User\SignupRequest;
use App\Http\Resources\User\UserResource;
use App\Models\User;
use Illuminate\Http\Request;
use Illuminate\Support\Facades\Hash;
use Illuminate\Validation\ValidationException;

class AuthController extends Controller
{
    function signup(SignupRequest $request)
    {
        $user = User::create([
            'name' => $request->name,
            'email' => $request->email,
            'password' => $request->password,
        ]);

        return response([
            'message' => 'User signed up.',
            'user' => new UserResource($user)
        ], 201);
    }

    function signin(SigninRequest $request)
    {
        $user = User::where('email', $request->email)->first();

        if (!Hash::check($request->password, $user->password)) {
            throw ValidationException::withMessages([
                'password' => 'Password does not match.',
            ]);
        }

        $token = $user->createToken('auth_token')->plainTextToken;

        return response([
            'message' => 'User signed in.',
            'user' => new UserResource($user),
            'token' => $token
        ], 200);
    }

    function signout(Request $request)
    {
        $user = $request->user();

        // option 1
        $user->currentAccessToken()->delete();

        // option 2
        $currentToken = $user->currentAccessToken();
        $user->tokens()->where('id', $currentToken->id)->delete();

        return response([
            'message' => 'User signed out.'
        ], 200);
    }

    function verify(Request $request)
    {
        return response([
            'message' => 'Token is valid.',
            'user' => new UserResource($request->user())
        ], 200);
    }
}
```

## How Each File Works

### Form Requests — SignupRequest & SigninRequest

Laravel Form Requests are dedicated classes that encapsulate two responsibilities that would otherwise live inside the controller: **authorization** and **validation**.

When a controller method type-hints a Form Request (e.g., `signup(SignupRequest $request)`), Laravel's service container resolves and runs the Form Request **before** the controller method body executes. The flow is:

1. The incoming HTTP request is bound to the Form Request class.
2. `authorize()` is called. Returning `false` immediately aborts with `403 Forbidden`. Returning `true` means any caller is allowed — appropriate here because `/api/signup` and `/api/signin` are open routes.
3. `rules()` is called. Laravel validates the request data against the returned rule array. On failure, it automatically returns a `422 Unprocessable Entity` JSON response with a structured `errors` object — no `try/catch` needed in the controller.
4. If both steps pass, the controller method receives a fully-validated request object.

**`SignupRequest` rules:**

| Field | Rules | Reason |
|---|---|---|
| `name` | `required\|string\|max:100` | Must be present, a string, no longer than 100 characters |
| `email` | `required\|email\|unique:users,email` | Must be a valid email address that does not already exist in the `users` table |
| `password` | `required\|string\|min:6\|max:10\|confirmed` | Must be 6–10 characters and match `password_confirmation` in the request body |

**`SigninRequest` rules:**

| Field | Rules | Reason |
|---|---|---|
| `email` | `required\|email\|exists:users,email` | Must be present, a valid email format, and exist in the `users` table |
| `password` | `required\|string\|min:6\|max:10` | Must meet the same length constraints as signup; `confirmed` is not needed here |

The `confirmed` rule on signup requires the request to also include a `password_confirmation` field with an identical value. This is a standard double-entry check for registration flows to catch typos. It is intentionally absent from signin — the user is proving knowledge of an existing password, not setting a new one.

### API Resource — UserResource

`UserResource` extends `JsonResource`, which is Laravel's built-in transformation layer for Eloquent models (and other data) before they are serialised to JSON.

**Why use a Resource instead of returning `$user` directly?**

When you return a raw Eloquent model from a controller, Laravel serialises every attribute in the model's `$visible` (or non-`$hidden`) set. In this project the `#[Hidden]` attribute on `User` already strips `password` and `remember_token`, but a Resource is more explicit and composable:

- It lists *exactly* which fields go into the response — no implicit leakage from future column additions.
- It is a single place to rename, format, or conditionally include fields.
- It can be reused across multiple endpoints without duplicating the shape definition.

**`toArray()` output:**

```json
{
    "id": 1,
    "name": "Alice",
    "email": "alice@example.com"
}
```

`$this->id`, `$this->name`, and `$this->email` delegate to the underlying `User` model's attributes via `JsonResource`'s magic `__get`. Only these three fields are included — `password`, `email_verified_at`, `created_at`, and `updated_at` are all omitted.

When `UserResource` is used inside `response([...])`, Laravel's JSON serialisation calls `toArray()` on it automatically. The resource wraps itself in a `data` key only when returned as the primary response (via `return new UserResource($user)`). When embedded inside an array (as done here with `'user' => new UserResource($user)`), the wrapping is transparent and the output is the plain array returned by `toArray()`.

### AuthController — updated

The controller is now a thin orchestrator. All input validation has moved into the Form Request classes; all output shaping has moved into `UserResource`.

**`signup(SignupRequest $request)`**

- The `$request->validate([...])` block is removed. Validation runs before this method is reached.
- `User::create()` and the `201` response remain unchanged.
- `$user` is now wrapped with `new UserResource($user)` instead of being passed directly.

**`signin(SigninRequest $request)`**

- The `$request->validate([...])` block is removed. Same reasoning as above.
- The credential check (`!$user || !Hash::check(...)`) and token creation are unchanged.
- `$user` in the response is wrapped with `new UserResource($user)`.

**`signout(Request $request)`**

- This method still uses `Illuminate\Http\Request` because the route is protected by `auth:sanctum` middleware and no body validation is needed — the user is identified from the Bearer token, not from request fields.
- The method body is unchanged from Session 2.1.
